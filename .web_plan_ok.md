### 1. 理解确认

你的目标是：把现有 Obsidian 插件 **Source Mode Inline Images** 发布到 Obsidian 官方社区插件目录，让用户可以在 Obsidian 的 Community Plugins 中搜索、安装和更新它。插件的核心能力是在 **Source mode** 中把 `![[image.png]]` 这类图片链接实时渲染为图片预览，并且不干扰 Live Preview 与 Reading View。

我需要先纠正一个可能过时的前提：当前官方文档描述的首次提交入口是 **community.obsidian.md**，而不是手动向 `obsidianmd/obsidian-releases` 提交 PR；不过仍然必须准备公开 GitHub 仓库、根目录 `manifest.json`、`README.md`、`LICENSE`，并创建带 `main.js`、`manifest.json`、`styles.css` 附件的 GitHub Release。官方目录会读取仓库默认分支的 `manifest.json`，安装时会从匹配版本号的 GitHub Release 下载插件文件。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submit your plugin.md))

------

### 2. 假设与前提

1. **插件源码可开源发布**。官方政策要求插件遵守版权与许可要求，并包含 `LICENSE` 文件；如果闭源，需要在 README 明确披露且可能按个案处理。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Developer policies.md))
2. **插件不需要 Node.js 或 Electron API**。从你提供的 `main.ts` 看，插件主要使用 Obsidian API、CodeMirror ViewPlugin、DOM API 和 `vault.getResourcePath()`，没有直接使用 `fs`、`crypto`、`os`、`electron` 等桌面专用 API，因此 `isDesktopOnly: false` 是合理的初始设定，但必须用 Android 和 iOS 实机验证。 官方要求：如果使用 Node.js 或 Electron API，`isDesktopOnly` 必须设为 `true`。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submission requirements for plugins.md))
3. **当前插件 ID 暂定为 `source-mode-inline-images`**。它符合“小写字母和连字符”的形式，且不包含 `obsidian`，也不以 `plugin` 结尾；这些都是官方 manifest 对插件 ID 的要求。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Reference/Manifest.md)) 但实施时仍要在社区目录或 `community-plugins.json` 中再次确认唯一性。
4. **插件当前版本 `0.1.0` 可以作为预发布感较强的初始版本**。官方要求版本号使用 `x.y.z` 的 SemVer 格式，`0.1.0` 符合格式；如果你希望给用户“稳定可用”的预期，可以在首次提交前改为 `1.0.0`。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submit your plugin.md))
5. **根目录 README 应使用英文，中文说明可以保留为补充文档**。官方要求仓库根目录有 README，说明插件用途和使用方式；社区插件详情页也会拉取仓库 README。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submit your plugin.md))
6. **GitHub Actions 不是硬性必须，但强烈建议添加**。官方文档明确给出了通过 GitHub Actions 自动构建并创建 Release 草稿的流程，且说明手动发布容易出错。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Release your plugin with GitHub Actions.md))
7. **`authorUrl` 建议改成 HTTPS**。你给出的 `http://k.appinn.me` 当前会跳转到 `https://k.appinn.me/qing/`，所以链接可访问，但 manifest 中最好直接写 HTTPS 的稳定作者主页或 GitHub profile，避免审核和用户信任问题。([Appinn](https://k.appinn.me/))

------

### 3. 总体思路

采用“**先修正审核风险，再补齐发布资产，最后提交目录审核**”的策略。

原因是 Obsidian 插件审核并不只是看功能是否可用，还会看 manifest、README、LICENSE、Release 附件、跨平台兼容性、资源清理、安全写法、样式可定制性等。官方插件指南特别提醒：避免 `innerHTML` 等不安全 DOM 写法、清理资源、避免直接访问 `workspace.activeLeaf`、避免硬编码样式等。你的代码整体功能清晰，但当前有两个比较容易被 reviewer 质疑的点：一是使用了 `plugin.app.workspace.activeLeaf` 判断源码模式，官方指南建议避免直接访问 `workspace.activeLeaf`；二是 `main.ts` 中直接写入了多处 inline style，官方指南建议优先使用 CSS class 和 CSS variables。 ([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Plugin guidelines.md))

因此不建议“直接创建 Release 然后提交”。更稳妥的路线是：

1. 先做一次面向官方审核的代码与元数据整理；
2. 准备英文 README、LICENSE、versions.json、CHANGELOG；
3. 用 GitHub Actions 固化构建与 Release 附件流程；
4. 在桌面和移动端完成验收；
5. 通过 `community.obsidian.md` 提交，或在平台要求时按其指引处理自动生成/关联的 PR。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submit your plugin.md))

#### 步骤 4：做一次审核导向的代码整理

- **做什么**：降低官方 reviewer 提出修改意见的概率。
- **在哪里**：`main.ts`。
- **怎么做**：

1. **避免直接使用 `workspace.activeLeaf`**

当前代码通过 `plugin.app.workspace.activeLeaf` 和 `getViewState()` 判断 Source mode。 官方指南建议避免直接访问 `workspace.activeLeaf`，优先使用 `getActiveViewOfType()` 或 `activeEditor`。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Plugin guidelines.md))

规划实现方向：

```ts
import { MarkdownView } from "obsidian";

function isStrictSourceMode(plugin: SourceModeInlineImagesPlugin): boolean {
  const markdownView = plugin.app.workspace.getActiveViewOfType(MarkdownView);
  if (!markdownView) return false;

  const state = markdownView.getState?.();
  return state?.source === true;
}
```

如果实施者发现这个判断无法准确区分多个编辑器窗格，则需要进一步测试并记录原因；不要为了“看起来符合指南”而引入错误行为。低置信度点：Obsidian 对编辑器内部状态的公开 API 有限制，多窗格场景可能需要额外验证。

1. **把可迁移的 inline style 移到 CSS**

当前 `ImageWidget.toDOM()` 中直接设置了 `img.style.width`、`maxWidth`、`height`、`display`、`margin` 等。 官方指南建议避免硬编码样式，尽量使用 CSS class 和 Obsidian CSS variables。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Plugin guidelines.md))

规划实现方向：

```ts
if (this.width) {
  wrapper.style.setProperty("--source-mode-inline-image-width", `${this.width}px`);
  wrapper.addClass("source-mode-inline-image-wrapper--custom-width");
}
```

然后在 `styles.css` 中控制：

```css
.source-mode-inline-image {
  display: block;
  max-width: min(480px, 100%);
  height: auto;
  object-fit: contain;
  border-radius: 4px;
  margin: 8px 0;
}

.source-mode-inline-image-wrapper--custom-width .source-mode-inline-image {
  width: var(--source-mode-inline-image-width);
  max-width: min(var(--source-mode-inline-image-width), 100%);
}
```

1. **把中文错误文案改为英文或双语**

当前找不到图片时显示中文错误。 建议改为：

```ts
`Image not found: ${imagePath}`
```

或：

```ts
`Image not found: ${imagePath} / 找不到图片文件：${imagePath}`
```

更推荐英文，因为官方目录和 README 建议面向国际用户。

1. **限制文件类型**

增加图片扩展名判断，避免非图片文件被塞进 `<img>`：

```ts
const IMAGE_EXTENSIONS = new Set([
  "png", "jpg", "jpeg", "gif", "webp", "bmp", "svg", "avif"
]);

function isImageFile(file: TFile): boolean {
  return IMAGE_EXTENSIONS.has(file.extension.toLowerCase());
}
```

1. **记录不支持的语法范围**

当前正则主要支持 `![[image.png]]` 和 `![[image.png|300]]`。如果暂不支持 `![[image.png|caption]]`、`![[image.png|300x200]]`、`![[image.png#fragment]]`，应在 README 明确说明，避免用户误解。

- **为什么**：这些改动不是功能扩展，而是提高审核通过率、跨主题兼容性、维护性和用户预期一致性。

#### 步骤 10：添加 GitHub Actions Release 工作流

- **做什么**：自动构建并创建 GitHub Release 草稿。
- **在哪里**：`.github/workflows/release.yml`。
- **怎么做**：基于官方示例：

```yml
name: Release Obsidian plugin

on:
  push:
    tags:
      - "*"

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: write

    steps:
      - uses: actions/checkout@v4

      - name: Use Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "18.x"

      - name: Install dependencies
        run: npm ci

      - name: Build plugin
        run: npm run build

      - name: Create release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          tag="${GITHUB_REF#refs/tags/}"
          gh release create "$tag" \
            --title="$tag" \
            --draft \
            main.js manifest.json styles.css
```

- **为什么**：官方文档说明手动发布耗时且容易出错，并给出 tag 触发、构建、上传 `main.js`、`manifest.json`、`styles.css` 的 Release 工作流。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Release your plugin with GitHub Actions.md))

    6. 创建新 tag 和 Release；
    7. 回到审核页面重新触发检查或等待复检。
  - 不要覆盖旧 Release 附件来“偷偷修复”，这样用户和审核状态容易混乱。
- **为什么**：官方文档说明，处理反馈时应更新仓库并发布递增版本的新 GitHub Release；插件只有在自动审核通过后才会在 Obsidian 内可安装。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submit your plugin.md))

------

#### 步骤 15：发布后的更新流程

- **做什么**：建立长期维护流程。
- **在哪里**：GitHub、`manifest.json`、`versions.json`、Release。
- **怎么做**：
  - Bug fix：`0.1.0 → 0.1.1`
  - 新功能：`0.1.0 → 0.2.0`
  - 破坏性变化：`1.x.x → 2.0.0`
  - 每次发布：
    1. 更新 `manifest.json` 的 `version`；
    2. 如果最低 Obsidian 版本变化，更新 `minAppVersion`；
    3. 更新 `versions.json`；
    4. 更新 `CHANGELOG.md`；
    5. 创建同名 tag；
    6. 发布带三件套附件的 GitHub Release。
- **为什么**：Obsidian 通过 Release 版本和 manifest 版本识别更新，`versions.json` 用于兼容旧 Obsidian 版本。([GitHub](https://github.com/obsidianmd/obsidian-sample-plugin))

### 5. 风险与应对

1. **风险：官方拒绝插件名称或 ID**
   - **表现**：ID 冲突、名称不符合规则、名称太像核心功能。
   - **应对**：提交前搜索社区目录；如果冲突，优先改名为更具体的名称，例如 `Source Mode Image Previews`，ID 相应改为 `source-mode-image-previews`。插件 ID 一旦发布后不宜再改。
2. **风险：`workspace.activeLeaf` 被 reviewer 要求修改**
   - **表现**：审核指出直接访问 active leaf。
   - **应对**：按步骤 4 改为 `getActiveViewOfType(MarkdownView)` 或 `activeEditor` 方向；如由于 CodeMirror ViewPlugin 限制必须使用内部状态，应在代码注释和 PR/提交说明中解释原因，并尽量隔离在一个函数中。官方指南明确建议避免直接访问 `workspace.activeLeaf`。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Plugin guidelines.md))
3. **风险：硬编码样式不符合主题兼容**
   - **表现**：reviewer 要求移除 inline style。
   - **应对**：把静态样式迁移到 `styles.css`，动态宽度用 CSS custom property。官方指南建议使用 CSS class 和 CSS variables。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Plugin guidelines.md))
4. **风险：移动端加载失败**
   - **表现**：Android/iOS 插件启用时报错，或图片路径无法渲染。
   - **应对**：如果是 API 不兼容且无法修复，必须把 `isDesktopOnly` 改为 `true`；如果只是样式或资源路径问题，则修复并保留移动端支持。官方要求使用桌面专用 API 时必须标记 desktop only。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submission requirements for plugins.md))
5. **风险：Release tag 与 manifest version 不一致**
   - **表现**：审核报找不到对应 Release，用户无法安装。
   - **应对**：使用 GitHub Actions，以 tag 触发 Release；tag 必须与 `manifest.json` 的 `version` 完全一致，不要加 `v` 前缀。([GitHub](https://github.com/obsidianmd/obsidian-sample-plugin))
6. **风险：README 不足**
   - **表现**：审核认为用途、安装方式、隐私说明不清楚。
   - **应对**：README 必须包含 purpose、features、usage、manual install、privacy、limitations、development、license。官方提交前要求 README 描述插件目的和使用方式。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submit your plugin.md))
7. **风险：中文文案影响国际用户理解**
   - **表现**：市场页面或错误提示对非中文用户不友好。
   - **应对**：根 README、manifest description、错误提示优先英文；中文说明作为 `README.zh-CN.md`。
8. **风险：误触 Live Preview 或 Reading View**
   - **表现**：Live Preview 中出现重复图片。
   - **应对**：把“严格 Source mode”作为核心测试项；多窗格、切换模式、重载插件都要测。
9. **风险：图片链接解析范围过窄**
   - **表现**：用户报告 caption、路径空格、子目录、特殊字符不工作。
   - **应对**：首版 README 明确支持范围；后续版本再扩展 parser。不要在描述里承诺未实现语法。
10. **风险：供应链或依赖问题**
    - **表现**：依赖过多、构建不可复现、reviewer 对依赖提出疑问。
    - **应对**：保持依赖最少；使用 `package-lock.json` 和 `npm ci`；Release 附件只上传构建产物三件套。

### 6. 验收标准

#### 元数据验收

- `manifest.json` 包含必填字段：`id`、`name`、`version`、`minAppVersion`、`description`、`author`、`isDesktopOnly`。
- `id` 只含小写字母和连字符，不包含 `obsidian`，不以 `plugin` 结尾。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Reference/Manifest.md))
- `name` 简短、英文、Basic Latin 字符，不包含 “Obsidian”。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Reference/Manifest.md))
- `description` 少于 250 字符，以句号结尾。([GitHub](https://raw.githubusercontent.com/obsidianmd/obsidian-developer-docs/main/en/Plugins/Releasing/Submission requirements for plugins.md))
- `version` 是 `x.y.z`。
- `authorUrl` 是 HTTPS 且可访问。
- `versions.json` 包含当前版本到 `minAppVersion` 的映射。

#### 仓库验收

- 根目录存在：
  - `README.md`
  - `LICENSE`
  - `manifest.json`
  - `main.ts`
  - `main.js`
  - `styles.css`
  - `package.json`
  - `package-lock.json`
  - `versions.json`
- README 英文说明完整。
- 中文说明作为补充文档存在。
- LICENSE 明确。
- 没有无关 sample plugin 残留代码、示例命名或未使用文件。

#### 构建验收

- `npm ci` 成功。
- `npm run build` 成功。
- 生成的 `main.js` 是最新产物。
- `main.js` 不依赖未打包的第三方运行时代码，除 Obsidian/CodeMirror 这类运行环境提供的外部模块外。
- Release 构建不包含 sourcemap，除非你有意公开 sourcemap。

#### 功能验收

在 Source mode：

- `![[test.png]]` 下方显示图片。
- `![[test.png|300]]` 显示为约 300px 宽。
- 子目录图片可显示，例如 `![[assets/test.png]]`。
- 文件名含空格时可显示，例如 `![[my image.png]]`。
- 找不到文件时显示友好错误。
- 非图片文件不会造成控制台异常。
- 长文档滚动时不明显卡顿。

在 Live Preview 和 Reading View：

- 插件不重复渲染图片。
- 不覆盖 Obsidian 原生图片行为。
- 切换模式后表现正确。

跨平台：

- Windows 测试通过。
- macOS 测试通过。
- Linux 尽量测试；如果无法测试，要在提交说明中诚实说明。
- Android 测试通过，否则不能声称移动端支持。
- iOS 测试通过，否则不能声称移动端支持。

#### Release 验收

- GitHub Release tag 与 `manifest.json.version` 完全一致。
- Release 附件单独包含：
  - `main.js`
  - `manifest.json`
  - `styles.css`
- 不是只依赖 GitHub 自动生成的 source zip/tar.gz。
- Release notes 简要说明首版功能和限制。
- 仓库默认分支上的 `manifest.json` 与 Release 附件中的 `manifest.json` 一致。

#### 官方提交验收

- 能在 `community.obsidian.md` 成功提交仓库 URL。
- 自动检查没有报：
  - manifest 不合法；
  - Release 不存在；
  - Release 附件缺失；
  - version/tag 不一致；
  - README 缺失；
  - LICENSE 缺失；
  - description 不合规。
- 如果收到审核反馈，按“新版本号 + 新 Release”的方式修复，而不是覆盖旧版本。
